## Context

Мастер агент обрабатывает изображения через `MediaClient` (Vertex AI). Изображения принимаются как base64 через REST API, обрабатываются в памяти и нигде не сохраняются. Проект уже работает в Google Cloud (Cloud Run, Vertex AI, Secret Manager, Firestore) — добавление GCS минимально по операционной сложности.

Бакет `master-agent-images` уже создан с папками `upload/` и `processed/`.

## Goals / Non-Goals

**Goals:**
- Сохранять каждое входящее изображение в `upload/` до обработки
- Сохранять обработанное изображение в `processed/` (если модель вернула изображение)
- Инкапсулировать GCS-логику в отдельный модуль `agent/gcs_client.py`
- Не блокировать ответ пользователю при сбое загрузки в GCS (fire-and-forget)

**Non-Goals:**
- Отдавать изображения из GCS через API (только запись)
- Управление жизненным циклом объектов (TTL, удаление)
- Генерация signed URLs для доступа к изображениям
- Сохранение аудио-файлов (только изображения)

## Decisions

### 1. Отдельный модуль `agent/gcs_client.py`

**Выбор:** Новый класс `GCSStorageClient` с методами `upload_original()` и `upload_processed()`.

**Почему не встроить в `MediaClient`:** `MediaClient` отвечает за взаимодействие с Vertex AI. GCS — другая ответственность. Разделение упрощает тестирование и замену реализации.

**Альтернатива:** Добавить методы напрямую в `processor.py` — отклонено, так как смешивает бизнес-логику с инфраструктурой.

---

### 2. Именование объектов в GCS

**Схема:** `{folder}/{session_id}/{unix_timestamp_ms}.{ext}`

Пример:
- `upload/user-123/1700000000000.jpg`
- `processed/user-123/1700000000000.png`

**Почему:** `session_id` группирует файлы по пользователю/разговору; timestamp гарантирует уникальность без UUID-зависимости.

---

### 3. Обработка ошибок GCS — fire-and-forget

**Выбор:** Ошибки загрузки в GCS логируются, но не пробрасываются. Ответ пользователю не зависит от GCS.

**Почему:** GCS — вторичная операция. Недоступность Storage не должна ломать основную функциональность агента.

**Альтернатива:** Строгий режим (пробрасывать ошибку) — отклонено, избыточно для аудит-хранилища.

---

### 4. Инициализация клиента

**Выбор:** `GCSStorageClient` инициализируется в `app.py` и передаётся в `MessageProcessor` как опциональная зависимость (аналогично `media_client`).

Если `GCS_BUCKET_NAME` не задан — клиент не создаётся, загрузка пропускается молча.

---

### 5. Синхронный vs асинхронный клиент

**Выбор:** `google-cloud-storage` — синхронная библиотека. Вызовы оборачиваются в `asyncio.get_event_loop().run_in_executor(None, ...)` чтобы не блокировать event loop FastAPI.

**Альтернатива:** `gcloud-aio-storage` (async) — отклонено, лишняя зависимость при наличии стандартного SDK.

## Risks / Trade-offs

- **[Риск] Задержка загрузки блокирует event loop** → Митигация: `run_in_executor` для синхронных GCS вызовов
- **[Риск] IAM не настроен на Cloud Run SA** → Митигация: задокументировать требование в tasks.md; при отсутствии прав клиент упадёт с PermissionDenied, которое логируется и не пробрасывается
- **[Trade-off] Fire-and-forget** — при сбое GCS файл теряется навсегда; приемлемо для аудита, неприемлемо если GCS — источник истины
